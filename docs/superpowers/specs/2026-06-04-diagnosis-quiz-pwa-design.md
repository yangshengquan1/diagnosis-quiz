# 执业医题眼诊断练习 PWA 设计稿

## 1. 目标

把当前的静态刷题页升级为一个可长期在线访问、手机和电脑都能打开、可分享给他人、可安装到主屏幕的公开网页应用。

本次设计优先解决以下问题：

- 当前访问方式不稳定，旧的临时外链不可持续。
- 直接双击 `index.html` 的打开方式不可靠，不适合作为正式使用方式。
- 用户需要一个关掉电脑后仍可访问的固定网址。
- 用户希望别人也能直接通过链接访问。

## 2. 非目标

本次不做以下能力：

- 用户登录
- 跨设备同步进度
- 服务端数据库
- 私有访问控制
- 原生 Android、iOS、Windows 打包应用

这些能力都需要后端或额外发布链路，不适合作为第一阶段上线目标。

## 3. 推荐方案

采用 `GitHub Pages + PWA` 方案。

理由：

- 当前项目是纯前端静态站点，天然适合 GitHub Pages。
- 不需要电脑常开，也不需要额外服务器。
- 发布后可得到长期稳定的公开网址。
- 补齐 `manifest`、`service worker` 和图标后，可作为 PWA 安装到手机或桌面。
- 维护成本最低，适合题库类应用。

## 4. 用户体验设计

### 4.1 核心入口

保留现有三大入口：

- 分类练习
- 随机练习
- 错题本

### 4.2 设备体验

- 手机端优先优化，确保单手刷题舒服，按钮大、文字清楚、滚动顺畅。
- 电脑端保留更宽松的内容宽度，适合连续练习。
- 首屏直接进入可操作界面，不做宣传页或说明页。

### 4.3 安装体验

- 支持浏览器“添加到主屏幕”。
- 安装后显示应用名、图标、主题色和启动页。
- 使用独立应用显示模式，减少浏览器界面干扰。

### 4.4 离线与弱网

- 用户成功打开过一次后，应尽量缓存核心静态资源。
- 网络较差时，已缓存的页面、样式、脚本和题库仍尽量可用。
- 不保证首次访问离线可用；首次仍需要联网加载站点。

## 5. 数据与状态设计

### 5.1 本地状态

继续使用浏览器本地存储保存：

- 当前答题模式
- 已练题目
- 错题本
- 最近使用视图

### 5.2 同步策略

第一阶段不做云同步。

结果是：

- 手机和电脑的练习记录互不影响。
- 每个访客都会拥有自己的本地练习记录。
- 分享链接不会泄露某个用户的做题进度。

这是一个有意的取舍，用最小复杂度先换取稳定上线。

## 6. 技术设计

### 6.1 架构

继续保持纯静态前端架构：

- `index.html` 作为应用入口
- `src/` 负责应用逻辑与视图渲染
- `data/questions.js` 作为题库数据源
- `styles.css` 负责界面样式

不引入服务端和构建框架，尽量在现有结构上演进，降低变更风险。

### 6.2 PWA 文件

新增或完善以下交付物：

- `manifest.webmanifest`
- `service-worker.js`
- 应用图标资源
- 更明确的部署与使用文档

### 6.3 缓存策略

优先缓存：

- 首页
- 样式文件
- 主脚本模块
- 题库数据文件
- 图标与 manifest

采用适合静态站点的保守缓存策略，目标是稳定可用，不追求复杂的运行时缓存系统。

### 6.4 部署适配

需要确认并修正 GitHub Pages 下的路径与资源引用，保证：

- 根路径部署可用
- 仓库子路径部署也可用

如果当前工作流已经存在，则在此基础上补齐发布所需资源，不重复发明部署链路。

## 7. 需要修复的现状问题

### 7.1 打开方式误导

README 中“直接双击 `index.html` 即可打开”的说法不应继续作为正式推荐方式。

原因：

- 页面采用 ES module 脚本加载。
- 部分浏览器对 `file://` 方式加载模块支持不稳定或存在限制。
- 这与“长期公开可访问”的正式交付目标不一致。

正式文档应改为：

- 本地开发或预览时通过本地静态服务访问
- 正式使用时通过公开部署网址访问

### 7.2 发布说明不足

需要把“如何发布成固定网址”写清楚，包括：

- 仓库结构要求
- GitHub Pages 设置方式
- 发布成功后的网址格式
- 后续更新题库时的发布流程

## 8. 实施范围

第一阶段应完成以下内容：

1. 修正现有应用在手机和电脑端的交互与布局细节
2. 补齐 PWA 所需资源与注册逻辑
3. 提供安装体验
4. 提供基础离线缓存
5. 整理并修正文档
6. 确认 GitHub Pages 发布链路可用

## 9. 验收标准

上线完成后，应满足以下标准：

- 电脑浏览器可通过固定公网网址访问
- 手机浏览器可通过同一网址访问
- 关闭开发电脑后，网址仍然可正常打开
- 第三方拿到链接后可直接使用
- 页面在手机端排版清晰，无明显错位
- 应用支持添加到主屏幕
- 二次访问时基础资源可从缓存中恢复
- 本地练习记录可在同一设备同一浏览器中保留

## 10. 风险与取舍

### 10.1 进度不同步

由于不做后端，同一用户在不同设备上的记录不会同步。

这是可接受取舍，因为用户当前最核心的需求是：

- 长期在线
- 关电脑仍可访问
- 可以分享给别人

### 10.2 公开访问

由于站点公开部署，拿到链接的人都可以访问题库内容。

这与当前用户选择 `A` 的公开访问策略一致。

### 10.3 Pages 缓存更新

静态站点在更新时可能受到浏览器缓存影响，因此实现中需要注意资源版本更新与缓存刷新策略，避免用户看到旧内容。

## 11. 测试策略

实现时应至少覆盖以下验证：

- 现有单元测试继续通过
- 新增与 PWA 相关的关键测试或最低限度的行为验证
- 本地静态服务下的页面可正常打开
- 手机尺寸与桌面尺寸下的界面检查
- 发布后对公网地址进行真实访问验证

## 12. 交付结果

本阶段交付物应包括：

- 可公开访问的刷题应用
- 固定网址
- 可安装的 PWA
- 更新后的部署与使用文档

## 13. 后续扩展建议

如果第一阶段使用稳定，可在后续阶段考虑：

- 云端同步进度
- 题目搜索
- 按系统统计掌握情况
- 导入更多题库
- 自定义错题复习策略
